## 5. Circuit Breaker Pattern

Inspired by Netflix's Hystrix library and resilience patterns in microservices, this example implements a circuit breaker that protects services from cascading failures. The circuit breaker monitors failures in calls to remote services, transitions between closed (normal), open (blocking calls), and half-open (testing recovery) states, and provides fallback mechanisms and bulkheading. The implementation shows how to configure failure thresholds, timeout windows, and recovery strategies, while maintaining metrics for observability. It demonstrates exponential backoff, jitter in retries, and integration with rate limiters. This pattern is essential in distributed systems like microservice architectures, API gateways (Kong, Envoy), and cloud-native applications to prevent resource exhaustion and improve fault tolerance.

## 6. Distributed Rate Limiter

This example builds a distributed rate limiting system similar to those used by API gateways like Kong or cloud providers like AWS API Gateway, implementing the token bucket or leaky bucket algorithms across multiple servers. The implementation addresses challenges of coordinating rate limit decisions across nodes, handling clock drift, and managing state in distributed caches (Redis). It demonstrates sliding window counters, hierarchical rate limiting (per-user, per-API, per-region), and graceful degradation under high load. The example covers techniques like local counting with periodic synchronization, distributed counting with atomic operations, and handling hot-key problems. This is critical for protecting backend services from abuse, implementing fair usage policies, and complying with external API rate limits.

## 7. Gossip Protocol for Cluster Membership

Based on protocols used in Cassandra, Consul, and Amazon's Dynamo, this example implements a gossip-based membership and failure detection protocol. Nodes periodically exchange information about cluster state, detect failures through heartbeats and suspicion mechanisms, and propagate membership changes through infection-style spreading. The implementation demonstrates SWIM (Scalable Weakly-consistent Infection-style Membership), tunable failure detection sensitivity (to balance false positives and detection time), and how gossip achieves eventual consistency with bounded overhead. It shows handling network partitions, deliberate node removals versus failures, and integrating with anti-entropy for data synchronization. This pattern is fundamental for building self-organizing clusters in distributed databases, service meshes, and peer-to-peer systems.

## 8. Two-Phase Commit Protocol

This example implements the classic distributed transaction protocol used in systems like PostgreSQL distributed transactions and XA-compliant databases. It demonstrates coordinating atomic commits across multiple participants, handling coordinator and participant failures through timeouts and recovery logs, and the blocking nature of 2PC under certain failure scenarios. The implementation covers prepare and commit phases, voting mechanisms, and the trade-offs between consistency and availability. It shows why 2PC is often avoided in modern distributed systems (due to blocking and coordination overhead) and introduces alternatives like Saga pattern for long-running transactions. This is essential for understanding distributed transactions in traditional databases, microservice choreography, and the theoretical foundations of distributed consistency.

## 9. Consistent Hashing Ring

Inspired by Dynamo, Cassandra, and content delivery networks, this example implements consistent hashing for distributing data across nodes with minimal rebalancing during cluster changes. The implementation demonstrates virtual nodes to achieve better load distribution, handling node additions and removals with bounded data movement, and the relationship between hash ring position and replica placement. It covers consistent hashing variants (jump hash, rendezvous hashing), partition rebalancing strategies, and how to maintain hotspot resistance. The example shows applications in distributed caching (Memcached), sharding strategies in databases, and load balancing in service meshes. This is crucial for building scalable distributed systems that need to dynamically add or remove capacity.

## 10. Leader Election with Raft

This example implements a simplified version of the Raft consensus algorithm used in etcd, Consul, and CockroachDB for leader election and maintaining a replicated log. It demonstrates the three roles (follower, candidate, leader), election process with randomized timeouts, and how nodes reach agreement on log entries. The implementation covers term numbers for detecting stale leaders, log replication with majority quorums, and safety properties (leader completeness, state machine safety). It shows handling network partitions, split votes, and log divergence resolution. This pattern is fundamental for building fault-tolerant distributed systems that need a single source of truth, coordinating distributed databases, and managing cluster metadata in systems like Kubernetes.

## 11. CQRS and Event Sourcing

Based on patterns used in systems like Axon Framework and EventStore, this example implements Command Query Responsibility Segregation with event sourcing. Commands modify state and generate events, which are persisted in an append-only log, while queries read from materialized views built from event streams. The implementation demonstrates eventual consistency between write and read models, event versioning and schema evolution, and snapshotting for performance. It covers replay mechanisms for rebuilding state, handling idempotency in command processing, and projecting events into multiple read models. This pattern is used in domain-driven design, audit-heavy systems like financial applications, and microservice architectures where different services need different views of the same data.

## 13. Distributed Cache with Write-Through/Write-Behind

This example implements a distributed caching layer similar to Hazelcast or Redis Cluster, with configurable write-through and write-behind strategies to underlying data stores. It demonstrates cache coherence protocols, invalidation strategies (time-to-live, event-based), and managing stale reads. The implementation covers partitioning data across cache nodes, handling cache node failures with replication, and preventing cache stampede during failures or cold starts. It shows techniques like bloom filters for negative lookups, probabilistic data structures for approximations, and the trade-offs between consistency and performance. This pattern is critical in high-traffic web applications (Facebook's TAO, Twitter's cache layer), reducing database load, and improving read latency.

## 14. Service Discovery and Health Checking

Based on Consul, Netflix Eureka, and Kubernetes DNS, this example implements service discovery with continuous health checking. Services register themselves with metadata, clients discover service instances through DNS or API queries, and unhealthy instances are automatically removed from the pool. The implementation demonstrates active health checks (HTTP/TCP probes), passive health checks (circuit breaker integration), and handling service registration/deregistration. It covers service versioning for rolling deployments, load balancing integration, and the role of TTLs in DNS-based discovery. This pattern is fundamental for microservice architectures, container orchestration platforms, and dynamic cloud environments where service instances frequently change.

## 16. Replication with Primary-Backup

This example implements primary-backup replication similar to PostgreSQL streaming replication or MySQL replication. A primary node handles all writes and replicates changes to backup nodes, which can serve reads. The implementation demonstrates synchronous versus asynchronous replication trade-offs, handling replication lag, and promoting a backup to primary during failures. It covers binlog-based replication, write-ahead log (WAL) shipping, and logical versus physical replication. The example shows preventing split-brain with fencing, handling replica divergence, and the impact of network delays on synchronous replication performance. This is essential for database high availability, read scaling, and disaster recovery in systems like AWS RDS and Azure SQL.

## 17. Bloom Filter Service

Based on distributed set membership testing used in systems like BigTable and Cassandra, this example implements a Bloom filter service for efficient probabilistic queries. The service allows checking set membership with controllable false positive rates but no false negatives, using minimal space compared to storing full sets. The implementation demonstrates hash function selection, bit array management across distributed nodes, and partitioning strategies for very large sets. It covers counting Bloom filters for supporting deletions, scalable Bloom filters for growing sets, and applications like caching which keys don't exist (to avoid expensive lookups), network routing, and spell-checking. This pattern is crucial for optimizing read performance in distributed databases and reducing network traffic.

## 18. Idempotent API Gateway

This example implements an API gateway with idempotency guarantees, similar to Stripe's API design or AWS API Gateway. Clients include idempotency keys in requests, and the gateway ensures that retrying the same request produces the same result without duplicating side effects. The implementation demonstrates storing request fingerprints and responses, managing idempotency key expiration, and handling concurrent requests with the same key. It covers the interaction with distributed caches for idempotency state, preventing resource leaks from abandoned requests, and integrating with rate limiting. This pattern is essential for building reliable APIs in payment processing, order management systems, and any service where exactly-once semantics are required despite at-least-once delivery from retries.
